# SeeDrift — manual

## Overview

SeeDrift measures **on-sky drift** by **plate solving** each **LIGHT** frame in a batch. **Frames are chosen from your NINA session log**: lines like **Saved image to …** under `BaseImageData.SaveToDisk` give exact paths on disk (no recursive imaging-folder scan). The trace is **ΔRA / ΔDec in arcseconds** relative to the **first solved frame for that FITS target** (each `OBJECT` gets its own chart when a batch mixes targets).

It does **not** subscribe to live saves for plotting; work happens when **SeeDrift Stop** finishes (or when you click **Run report** on the plugin options page).

## Installation

See [README.md](../README.md). Copy **`NINA.Plugin.SeeDrift.dll`** (and dependencies from the same build output if the host asks).

## Options (Plugins → SeeDrift)

For **Discord webhook**, **Concurrency**, and **Minimum exposures**, hover anywhere on the row (label or field) to see the full hint in a tooltip.

| Setting | Meaning |
|--------|---------|
| **Discord webhook** | Optional **Discord Integrations → Webhooks** URL (`https://discord.com/api/webhooks/…` or legacy `discordapp.com`). After **SeeDrift Stop** saves night HTML, SeeDrift POSTs that file to the channel (multipart). If Stop finishes with **nothing chartable** (too few saved-light lines, LIGHT frames, or plate solves), SeeDrift still sends a **text-only** webhook explaining that outcome when the URL is set. **Run report** never uploads. The URL embeds a **secret** token — SeeDrift **never** writes it to the log. Empty = disabled. If the HTML exceeds Discord’s attachment limit (~25 MiB), SeeDrift sends a short text-only webhook message instead (full path is **not** echoed to Discord; check **`SeeDrift.log`** locally). Failures are log-only. |
| **Concurrency** | Choose concurrent plate solves from **1** up to **floor(80% × physical cores)** (minimum **1**). Physical cores come from **`GetLogicalProcessorInformation`** (`RelationProcessorCore` entries); if that API fails, SeeDrift uses **`Environment.ProcessorCount`** (logical processors) for both the percentage cap and the default. On a **new** install with no saved value, the default matches physical cores **clamped** to that ceiling (e.g. 12 cores → max **9**, default **9**). Each concurrent slot gets its **own** solver instance; higher values use more RAM and CPU. Use **1** to mimic older sequential behavior. Image **downsampling** for solves comes only from **NINA Options → Plate Solve** (same profile as the Plate Solve tool)—SeeDrift does not add a separate downsample step. |
| **Minimum exposures per target** | Each **Stop** or **Run report** batch can mix several FITS **OBJECT** names. SeeDrift lists **only** targets that have at least this many **solved** frames in that batch (1–500, default **50**). Targets below the threshold are omitted from the batch **heading**, drift charts, and sequencer blocks for that run. If **no** target meets the threshold, the batch section explains that instead of drawing empty charts. |
| **Alternative image location mapping** | Optional **Original image root** and **Alternative image root** (folder pickers; hover the section for the full explanation). SeeDrift plate-solves using each **Saved image to …** path from the log when that file still exists. When it is missing and the log path starts with **Original image root**, the same subfolders and filename are tried under **Alternative image root** (any NINA file-pattern layout under that prefix). Set original root to match your **logs**, not necessarily today’s NINA imaging path. Leave both empty to disable. |
| **Recent NINA logs** (under **Create report**) | Used only by **Run report**. SeeDrift scans `%LocalAppData%\NINA\Logs`, sorts by the log start time, and shows a compact dropdown with the **last ten** logs (detected Seestar identity, target count, usable image count, duration). Logs with zero detected targets are hidden from the dropdown. Selecting a row fills the persisted **Log file path**. **Browse** and paste remain available for older logs, startup-only logs, or custom paths. **Unknown scope** in the dropdown means SeeDrift did not find Seestar Alpaca discovery, Telephoto Camera **connect**, Telephoto **disconnect** (`Disconnecting from … Seestar … Telephoto Camera`), or matching lines in that log scan — it does not block solving. When the path matches a night HTML already in **`%LocalAppData%\NINA\SeeDrift\Reports`** (embedded log paths + frame/processing summary), the status panel shows **Open** and a short summary without re-running the report. The completion line and **Open** link always match the **currently selected** log (including after **SeeDrift Stop** runs that used different logs). Older HTML without embedded log paths may still match via the page header text. |
| **Compare saved reports** | Pick a **Before** and **After** SeeDrift HTML report created by a schema-compatible analytics build, then click **Compare saved reports**. The saved-report dropdown scans **`%LocalAppData%\NINA\SeeDrift\Reports`** and shows the report’s imaging session date and detected Seestar identity from embedded metadata; **Before** and **After** fill the path fields. **Delete selected** (after confirmation) removes **exactly one** library `.html` file—the row highlighted in the dropdown—each time; repeat to remove others. **Refresh** only rescans the folder and **never** deletes files. Browse/paste remain available for reports saved elsewhere. SeeDrift extracts the embedded report data and writes a versioned comparison HTML in the report library. Different plugin versions can be compared when the embedded report schema is supported; unsupported future schemas are rejected with a clear message. The comparison uses whole-report averages for dither RA/Dec behavior and center-after-drift recovery; it does **not** require matching target names, scan FITS files, or run plate solving. If before/after reports came from different Seestars, the comparison report shows an advisory to read arcsecond and pixel-scale-sensitive metrics cautiously. |

SeeDrift also appends its own messages to **`%LocalAppData%\NINA\SeeDrift\SeeDrift.log`** (same folder as `settings.json`). The main NINA application log still receives the same lines.

**Tip:** Plate solving uses your **NINA plate-solve profile** (same engines/settings as the Plate Solve tool). A typical ~300-frame session can take **10–20 minutes** depending on solver speed and your **Concurrency** setting. To reduce solve time, tune that profile (solver choice, timeouts, downsampling); then adjust **Concurrency** if your PC can sustain multiple solves. **Video previews** (when enabled) add extra build time on top of plate solving — each target's video encodes all frames with no sampling, so more frames = longer wait. The **Compare saved reports** path avoids solve and video costs entirely by using metrics already embedded in generated HTML.

## Workflow

### Automated window (sequencer)

**SeeDrift Start** and **SeeDrift Stop** use a small **SeeDrift** vector icon (axes + drift trace) in the sequencer instruction list and on each step block; the geometry is assigned in code so it always appears.

1. Insert **SeeDrift Start** so the plugin records **arm** time (UTC internally).
2. Capture lights; NINA writes **Saved image to …** lines into the active session log under **`%LocalAppData%\NINA\Logs`**.
3. Insert **SeeDrift Stop** — the plugin records **disarm** time, **pre-filters** NINA logs in that folder by the log file name’s local date around your **[arm, disarm]** window (with a safe fallback to reading **all** `*.log` files if none match), keeps **Saved image to …** lines whose timestamp falls in **[arm, disarm]** inclusive, collects FITS paths, skips calibration-looking frames when FITS keywords say so, plate-solves in log order, correlates **CenterAfterDrift** / **DitherAfterExposures** using **only logs that contributed those save lines**, and updates the **night HTML**.

### Create report (from a log)

1. Under **Create report**, choose one of the last ten logs from the dropdown (or pick any `.log` via Browse/paste when available).
2. Click **Run report**. SeeDrift collects **every** saved-light path from that file (full log — no Start/Stop window), then solves and correlates like Stop.

**While the run is in progress**, a **status panel** appears directly under **Run report** (reading log → checking FITS headers → solving each frame). When the run ends, the **last status line stays visible** (for example **Complete — …** with elapsed time) until you start another run or leave the panel cleared by starting again. **Open** shows the saved HTML **file name** as an underlined control—click it to open the report in your default browser (hover for the full path). Changing the **recent logs** dropdown or the path field refreshes **Open** to the newest matching report on disk for that `.log` when one exists (after a short debounce while typing a path). If nothing is added to the HTML, a dialog explains common causes. **SeeDrift Stop** uses **NINA’s main status bar** for progress and completion; that bar is plain text (not clickable). After Stop, open **Plugins → SeeDrift** to use **Open** for the same night HTML file when the run succeeded.

### Rolling HTML file

Each **SeeDrift Stop** or **Run report** completion is one **run**: it appends a new section to the rolling night report in **`%LocalAppData%\NINA\SeeDrift\Reports`** for the current plugin version/session date. (**Run report** reads exactly **one** `.log` file that you choose. **Stop** reads a **filtered** set of logs (by filename date around arm/disarm, with fallback to the full folder), then keeps only **Saved image to …** lines in your arm/disarm window.) SeeDrift also scans the contributing NINA log files for lines such as **`Discovered Alpaca Device Seestar S30_0ac17a9b Telescope`**, **`Successfully connected Camera`** with **`Seestar … Telephoto Camera`**, or (when you were already connected at startup and lack connect lines) **`Disconnecting from`** … **`Seestar … Telephoto Camera`** from ASCOM disconnect (compact id **`S30_0ac17a9b`**). When found, the compact scope identity is shown in recent-log and saved-report dropdowns, embedded in the report data, shown in the HTML header, and included in report filenames. If no matching Seestar line is present, UI labels show **Unknown scope** and filenames omit the scope token. The page header’s **NINA log files** list shows **only logs that actually contributed** those save lines for each run (not every file that was scanned). The page header shows the **same featured screenshot** bundled inside the plugin DLL (**embedded PNG**, inlined into the HTML as a **data URI** so it works **offline** with no external image fetch). If that embedded asset were ever missing from a custom build, SeeDrift falls back to **FeaturedImageURL** from assembly metadata, then the **`SeeDrift_Icon`** vector (`Resources.xaml`). The **Session Log** line uses the **session date** from NINA log file naming when possible (same basis as **`sess`** in the saved file name), not necessarily the day you generated the report. Header layout also includes the **generated** time, report version/schema, scope identity, log path(s), and how to **reset chart zoom**. For **Stop**, one run may reference **several** log files when NINA rotated logs mid-session; for **Run report**, one run references **one** file. When the HTML lists several paths, that is usually the **union** across multiple runs on the same calendar-night page. Under each **run** heading (targets summary), the next line is **solved frame count** and **processing time** for that run (log read through saving HTML). For each **Target** chart, **Start** and **End** (or a single **Exposure** line) show the first and last **exposure start** time among solved frames for that target, in **local** wall time (from FITS / the same times used to order frames). When a run spans **several** FITS **OBJECT** names, the **heading lists only targets that meet Minimum exposures per target**, in the same first-seen frame order as the **Target:** subsections below. The HTML shows **one drift chart and one Sequencer events block per listed target**; drift is zeroed at each target’s first solved frame, and sequencer rows only list intervals between **two consecutive frames of that target**. File names in rows remain the actual saved paths.

The report page loads **Tailwind CSS**, **Chart.js**, **Hammer.js**, and **chartjs-plugin-zoom** from CDNs (needs network the first time you open the file). Each drift chart uses the **same arcsecond span** on ΔRA and ΔDec (the wider axis range sets the span for both, centered on the data) and a **square** plot so **1″ horizontally equals 1″ vertically**. The drift chart **legend** lists the target trace only (not dither/center markers). Under each chart, the first line is **Σ|Δstep|** on each sky axis (**ΔRA …″ · ΔDec …″** vs the target’s first solved frame; optional detector or **≈** pixel equivalents as below). When logs place a **dither** between exposures, a bordered logged-dither block shows a bold **Logged dither intervals total (assessed) — Σ|ΔRA|** / **Σ|ΔDec|** line summed over **non-suspect** dither intervals only (and detector Σ|Δx|/Σ|Δy| when applicable). If any **suspect tracking jumps** were identified (same rule as the dither effectiveness score: leave-one-out vs other logged dithers on the target — vector |Δ| ≥ max(300″, 2.75× peer median), or |ΔRA|/|ΔDec| far outside peer medians on that axis, or measured move ≫ commanded guider pulse when logs include it), an **amber subtext** below the total lists how many intervals were excluded and the discounted Σ|ΔRA| / Σ|ΔDec| (and detector px) so the older "everything summed" number can still be reconstructed. A second **Typical dither** line shows the **median |Δ|** across assessed dithers (with detector px when the plate scale is known) so a single huge jump cannot hide the typical session behaviour. The individual **Exposures with logged dithers between them** results are in a collapsed-by-default table with each interval’s signed segment **ΔRA / ΔDec** (and **|Δx| / |Δy|** on those hops when the trace uses detector registration); suspect jumps are flagged in the table and excluded from dither assessment. **Pixel** line on the first row: if the run used **detector** cumulative registration, **Σ|Δx|** and **Σ|Δy|** are sums of absolute pixel steps; otherwise **≈** pixel totals divide those arcsecond sums by a **median** **″/px** from each frame’s FITS **XPIXSZ**+**FOCALLEN** (rough axis-aligned equivalent, not a full projection). The drift chart supports **wheel zoom** (including zooming **out** past the tight data bounds for extra context), **drag pan**, **double‑click** on the chart to reset zoom, and keyboard **R** or **Esc** to reset zoom on **all** drift charts on the page (ignored while typing in an input). The **ΔDec** axis is **reversed** vertically so motion on screen aligns with typical camera/imaging “up/down”; hover still shows the same numeric ΔRA/ΔDec as before. Points along the path show **start** (green) and **end** (orange); a single-frame target uses one **amber** marker. **Hover** on the path shows the **file name** and arcsecond offsets. When logs correlate **dither** or **center-after-drift** between two frames, **purple triangles** and **pink squares** appear **on the line segment between those frames** (midpoint for a single event; evenly spaced if several). **Dither** triangles **point along that segment toward the later frame**; center squares stay axis-aligned. When the same target is scheduled again later in the night, NINA may log **`TargetScheduler-NewTargetStart`** between two solved frames while exposure numbering continues globally. SeeDrift splits the target into **visits**: each visit gets its own drift trace (re-zeroed at that visit’s first frame; **no line** drawn across visits), with **Visit N** lines under the target heading for per-visit frame counts and exposure times while the first subtitle line still shows the **night-wide** total for that target. An amber **↻** on the gap segment marks a **return visit** (scheduler restart; not missing frames). When exposure numbers jump without that log signal, an amber **?** still marks **possibly missing or unsolved** frames with tooltip detail. Hover gap markers for the full explanation. The **Sequencer events** table wraps long **From/To** paths so the **Detail** column keeps space.

Each target also includes an **analytics** strip. **Drift / walking-noise risk** is an advisory color box that looks at intervals without logged dither or center commands. From 0.8.24 the advisory is split into two sub-tiers, each shown as its own chip, with the surrounding box tone driven by the worse of the two (Low / Moderate / Caution):

- **Star shape** — about per-exposure motion only. Thresholds are sized to the community "< 1–2 px per exposure is acceptable" rule of thumb: Moderate at `≥ 1.0 px` (or `≥ 2.0″`) per exposure, Caution at `≥ 2.5 px` (or `≥ 5.0″`). Highly directional drift can nudge the level up but only when the per-exposure motion also clears a small floor (`≥ 0.3 px` / `0.8″` for Low→Moderate, `≥ 0.5 px` / `1.5″` for Moderate→Caution), so a visually tame trace cannot flag Caution purely on consistency.
- **Walking noise** — about correlated FPN across the stacked image. **Dither headroom** = `median assessed dither |Δ| (px) / median cumulative drift between consecutive dithers (px)`, where the median dither comes from **plate-solve measured** moves at logged dither edges (same as the dither effectiveness table). Caution when `direction consistency ≥ 0.75 AND headroom < 1.5× AND between-dither drift ≥ 0.5 px`; Moderate when `consistency ≥ 0.5 AND headroom < 4× AND ≥ 0.25 px`; Low otherwise. From 0.8.31: with **≥ 2** assessed dithers, **&lt; 25%** direction consistency forces **Low** (zigzag drift is not walking noise). If headroom cannot be computed but assessed dithers exist, a short 5-frame burst must exceed **1.5×** the median assessed dither (and **≥ 40%** consistency for Caution) before the walking-noise tier escalates. With no logged dithers, the sub-tier falls back to worst short-window drift (5/10/15 frames), requiring **≥ 40%** consistency for a **≥ 5 px** burst to reach Caution. FITS plate scale and exposure duration sharpen both sub-tiers; everything remains advisory.

The hint lines under the chips now read:

- **Star-shape hint:** estimated `…″` drift during a typical exposure (`≈ … px`), with a reminder that round stars typically tolerate `< 1–2 px` of motion per exposure.
- **Walking-noise hint:** dither headroom `…×` (median dither magnitude vs cumulative drift between dithers — `…″ · ≈ … px between dithers`). Higher is safer. When no logged dithers were available, the hint falls back to the worst short-window drift line.

The advisory `Detail` line now names which signal drove the headline (e.g. *"Walking-noise signal drives the headline: dither headroom 1.5× with 87% directional drift between dithers."*), so a Caution headline never disagrees with what stars in single subs look like. **Recommendations** are rewritten around the two sub-tiers: star-shape Caution suggests shorter exposures and polar alignment / guiding; walking-noise Caution suggests increasing **Mount Dither Pixels** in NINA, dithering more often, and addressing field drift (polar alignment / differential flexure). When the walking-noise sub-tier is Caution but star-shape is Low, an info line is added so users do not chase a star-shape problem that is not there. If logged dithers repeated the same direction as the natural drift, an extra recommendation calls that out because randomization is not breaking the FPN pattern. A Moderate walking-noise verdict with `headroom < 3×` also emits an info recommendation that a larger dither widens the margin. **Drift rate** shows net arcseconds per minute (and an approximate pixel/min value when FITS plate scale is available). **Dither effectiveness** counts assessed logged dithers, median measured movement, weak dithers, and repeated-direction dithers. Very large outlier dither intervals are treated as **suspect tracking** and excluded from those dither statistics; the strip reports how much absolute RA/Dec was discounted. **Center-after-drift** counts center events as helpful, partial, or ineffective by comparing pre-center drift to residual drift over the next few frames. A compact **Session quality timeline** labels intervals as stable, drifting, dither recovery, or center recovery. **Settings hints from this session** are advisory only; they summarize patterns such as walking-noise drift risk, weak dithers, high drift rate, repeated dither direction, suspect tracking intervals, or center events that did not reduce residual drift. From 0.8.25, when the **median realized dither magnitude** (measured / commanded) is **below 80%** across **at least 3** samples, an additional warn recommendation calls the shortfall out. NINA runs the dither pulse to completion before the settle countdown starts, so this signal mostly surfaces mount mechanics — Dec backlash on direction-reversing dithers or a small guide-rate calibration offset on the Seestar Alpaca side are the typical culprits on small EQ mounts. A too-short settle time can still bias the measured centroid via motion blur during the next exposure, so settle is mentioned as a secondary suspect in the recommendation. "Weak" dither flags use the frame-to-frame noise floor as their reference; this new check uses the commanded slew directly, so it spots a different failure mode (the mount not delivering the commanded movement) rather than dithers that are simply too small.

New reports embed a structured JSON data block in the HTML. This is what **Compare saved reports** reads for before/after comparisons, so old reports without that block need to be regenerated before they can be compared.

### Session settings used (run-wide panel)

Near the top of each night HTML, above the per-target sections, a **Session settings used** card surfaces the NINA + Seestar values that actually drove this run. Values come from log lines that the correlator already reads, so the panel never opens additional files. Rows that have no observations are omitted (no empty zeros). The card is **collapsed by default** — click the header to expand it. The same panel inside **Compare saved reports** stays expanded inline so before / after rows are visible at a glance.

| Row | Source line | What it represents |
|---|---|---|
| **Mount Dither Device — Dither Pixels** | `DirectGuider.cs\|SelectDitherPulse\|...\|Dither target from (x0, y0) to (x1, y1)` | Median commanded `max(\|Δx\|, \|Δy\|)` across all pulses; a faithful proxy for the **Mount Dither Pixels** setting that NINA actually told the mount to use. From 0.8.25, when at least **3** assessed dither pairs have both a commanded magnitude (`√(Δx²+Δy²)` from the matched pulse line) and a measured magnitude (frame-to-frame Δ converted via the median plate scale), a sibling **Realized X.X px (Y%) median across N pulses** line is shown directly below this row. Run-wide values are the mean of the per-target medians (suspect dithers excluded). |
| **Center After Drift — Max arc-minutes** | `CenterAfterDriftTrigger.cs\|PlatesolvingImageFollower_PropertyChanged\|...\|Drift: X / Y arc minutes` | The Y value is the **threshold** configured on the CenterAfterDrift trigger. Variance across the run is flagged. |
| **Center After Drift — Evaluate after exposures** | Same evaluation lines, one per evaluation | Cadence is inferred from frame gaps between consecutive evaluation lines and labelled **inferred**. |
| **Dither After Exposures — Cadence** | `Starting Trigger: ... DitherAfterExposures` (parsed `AfterExposures = N` when the NINA build prints it), with frame-gap fallback | Either parsed from the log or inferred from gaps between successive triggers; inferred values are tagged. |
| **Dither pulse guide durations** | `using guide durations of X and Y seconds` on the dither pulse line | Median seconds across pulses (RA / Dec). The Seestar Alpaca RA/Dec guide rate is **not** in NINA logs; this row is the visible proxy. |

The same values are also embedded in the report's JSON payload (additive `SequencerSettings` field). **Compare saved reports** reads them and adds a **Session settings used** table to the comparison HTML — each row shows the **Before** and **After** value with a **Changed / Same / —** badge so you can tell at a glance whether metric improvements / regressions came alongside a configuration change. When any row is **Changed**, the comparison's overall summary appends a short tip pointing at that table. Reports created before this metadata existed still compare and simply show `—` for the settings rows.

The comparison metrics table additionally includes **Σ|ΔRA| over assessed dither intervals**, **Σ|ΔDec| over assessed dither intervals**, and **Typical dither (median |Δ|)** rows. These mirror the night-report totals (suspect intervals excluded) so a small or weak dither pattern is easy to spot across sessions. Older reports without per-target totals fall back to summing the embedded dither events directly, so comparison still works across plugin versions.

From 0.8.24 the metrics table also surfaces two walking-noise-specific rows pulled from each report's `DriftRisk`:

- **Dither vs drift headroom** (×, higher is better) — average headroom across targets in the report.
- **Worst short-window drift** (px, lower is better) — average of each target's worst 5/10/15-frame drift in pixels.

Together these let you see, side by side, whether changing **Mount Dither Pixels** or fixing field drift between sessions moved the actual walking-noise risk — not just the per-exposure or per-event averages. Older reports that pre-date these fields render `0` (no change shown).

### Sequencer events in HTML

When log lines match **between-frame** intervals, **dither** and **center-after-drift** triggers appear in **Sequencer events (NINA logs)** — same strict rules as before (paired save times when present, fractional seconds, etc.). Timestamps without `Z` are treated as **local wall time**. If nothing correlates, you still see an explanatory empty state. When rows exist, the table is **collapsed by default**; click the summary line to expand it.

## Technical notes

- **LIGHT filter:** After resolving paths from the log, SeeDrift reads each FITS/XISF primary header and skips frames whose **IMAGETYP** / **OBSTYPE** indicate calibration (bias/dark/flat) when those keywords are present — same as earlier builds. File extensions **.fits**, **.fit**, **.fts**, and **.xisf** are accepted from the log.
- **Minimum exposures gate:** After LIGHT frames are identified from FITS headers, SeeDrift checks whether **any single OBJECT** has at least **Minimum exposures per target** LIGHT frames in that run. If **none** do, it **does not plate-solve** (nothing useful would chart). During solving, if plate failures make it **impossible** for any OBJECT to reach that many **successful** solves (even if every remaining frame succeeded), SeeDrift **skips the remaining solves** for that run so you are not stuck solving hundreds of frames that cannot change the outcome. If solves finish (or stop early) but **no target** has enough **successful** solves for that threshold, SeeDrift **still skips correlating NINA sequencer logs** into those frames (saves time on large log sets). The night HTML and **Complete —** status explain both cases.
- **Stop vs Run report:** Stop restricts **Saved image to …** lines to **[arm, disarm]** by log timestamp. **Run report** uses **the entire chosen log file**.
- **Logs folder:** Stop works from **`%LocalAppData%\NINA\Logs\*.log`** only — fast compared to walking the image tree. It **pre-filters** by the `yyyyMMdd-HHmmss-…` prefix in each log file name (local dates around arm/disarm, plus one day before the earlier date). If **no** files match that filter, SeeDrift reads **all** logs and logs a warning (covers unusual file names). In practice you often get **one log file per imaging session**, but NINA may **start a new file** when it rotates logs, so one session can span **two** files. The night HTML lists **only** logs that had at least one in-window **Saved image to …** line for that run.
- **Solve failures:** Frames that do not solve are skipped for the trace (implementation logs errors).
- **Jump detection:** Runs on the solved-sample sequence where applicable.
- **Video preview speed:** When **Auto-generate video** is enabled, each target encodes **all** frames using embedded FFmpeg — no frame sampling. A target with ~200 frames at 480p takes roughly 30–60 seconds to encode. At higher resolutions or with the drift reticle enabled (frame-by-frame overlay drawing), expect longer build times.

## Troubleshooting

| Symptom | Likely cause |
|--------|----------------|
| Empty or tiny trace | Fewer than two **Saved image to …** lines in range (Stop) or in the file (Run report); paths missing on disk; plate solve failing — check logs and solver profile. |
| “Done” but no HTML where I looked | Reports are stored in **`%LocalAppData%\NINA\SeeDrift\Reports`**. Copy the path from **NINA’s status bar** after a successful run, use the **Open** link in Plugins → SeeDrift, or check **`%LocalAppData%\NINA\SeeDrift\SeeDrift.log`** if the status line reports a save error. |
| Paths not found | Log contains paths from another PC or moved folders — files must exist where the log says. |
| No sequencer table | Triggers not in strict between-frame intervals for this ordering — expected sometimes. |
| Plugin DLL not updating | NINA has the DLL locked — close NINA and rebuild/copy. |
