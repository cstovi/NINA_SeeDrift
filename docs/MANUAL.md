# SeeDrift ŌĆö manual

## Overview

SeeDrift measures **on-sky drift** by **plate solving** each **LIGHT** frame in a batch. **Frames are chosen from your NINA session log**: lines like **Saved image to ŌĆ”** under `BaseImageData.SaveToDisk` give exact paths on disk (no recursive imaging-folder scan). The trace is **╬öRA / ╬öDec in arcseconds** relative to the **first solved frame for that FITS target** (each `OBJECT` gets its own chart when a batch mixes targets).

It does **not** subscribe to live saves for plotting; work happens when **SeeDrift Stop** finishes (or when you run **Previous session report** from plugin options).

## Installation

See [README.md](../README.md). Copy **`NINA.Plugin.SeeDrift.dll`** (and dependencies from the same build output if the host asks).

## Workflow

### Automated window (sequencer)

**SeeDrift Start** and **SeeDrift Stop** use a small **SeeDrift** vector icon (axes + drift trace) in the sequencer instruction list and on each step block; the geometry is assigned in code so it always appears.

1. Insert **SeeDrift Start** so the plugin records **arm** time (UTC internally).
2. Capture lights; NINA writes **Saved image to ŌĆ”** lines into the active session log under **`%LocalAppData%\NINA\Logs`**.
3. Insert **SeeDrift Stop** ŌĆö the plugin records **disarm** time, **pre-filters** NINA logs in that folder by the log file nameŌĆÖs local date around your **[arm, disarm]** window (with a safe fallback to reading **all** `*.log` files if none match), keeps **Saved image to ŌĆ”** lines whose timestamp falls in **[arm, disarm]** inclusive, collects FITS paths, skips calibration-looking frames when FITS keywords say so, plate-solves in log order, correlates **CenterAfterDrift** / **DitherAfterExposures** using **only logs that contributed those save lines**, and updates the **night HTML**.

### Previous session (pick a log file)

1. Under **Previous session Log**, choose a recent log from the last-14-days dropdown, or Browse/paste a full path to a `.log` file.
2. Click **Run previous session report**. SeeDrift collects **every** saved-light path from that file (full file, no time picker), then solves and correlates like Stop.

**While the run is in progress**, a **status panel** appears directly under **Run previous session report** (reading log ŌåÆ checking FITS headers ŌåÆ solving each frame). When the run ends, the **last status line stays visible** (for example **Complete ŌĆö ŌĆ”** with elapsed time) until you start another run or leave the panel cleared by starting again. **Open** shows the saved HTML **file name** as an underlined controlŌĆöclick it to open the report in your default browser (hover for the full path). Changing the **recent logs** dropdown or the path field refreshes **Open** to the newest matching report on disk for that `.log` when one exists (after a short debounce while typing a path). If nothing is added to the HTML, a dialog explains common causes. **SeeDrift Stop** uses **NINAŌĆÖs main status bar** for progress and completion; that bar is plain text (not clickable). After Stop, open **Plugins ŌåÆ SeeDrift** to use **Open** for the same night HTML file when the run succeeded.

(**Previous session report** reads exactly **one** `.log` file that you choose. **Stop** reads a **filtered** set of logs (by filename date around arm/disarm, with fallback to the full folder), then keeps only **Saved image to ŌĆ”** lines in your arm/disarm window.) When found, the compact scope identity is shown in recent-log and saved-report dropdowns, embedded in the report data, shown in the HTML header, and included in report filenames. The page headerŌĆÖs **NINA log files** list shows **only logs that actually contributed** those save lines for each run (not every file that was scanned). The page header shows the **same featured screenshot** bundled inside the plugin DLL (**embedded PNG**, inlined into the HTML as a **data URI** so it works **offline** with no external image fetch). When the HTML lists several paths, that is usually the **union** across multiple runs on the same calendar-night page. Under each **run** heading (targets summary), the next line is **solved frame count** and **processing time** for that run (log read through saving HTML). For each **Target** chart, **Start** and **End** (or a single **Exposure** line) show the first and last **exposure start** time among solved frames for that target, in **local** wall time (from FITS / the same times used to order frames). When a run spans **several** FITS **OBJECT** names, the **heading lists only targets that meet Minimum exposures per target**, in the same first-seen frame order as the **Target:** subsections below. The HTML shows **one drift chart and one Sequencer events block per listed target**; drift is zeroed at each targetŌĆÖs first solved frame, and sequencer rows only list intervals between **two consecutive frames of that target**. File names in rows remain the actual saved paths.

The report page loads **Tailwind CSS**, **Chart.js**, **Hammer.js**, and **chartjs-plugin-zoom** from CDNs (needs network the first time you open the file). Each drift chart uses the **same arcsecond span** on ╬öRA and ╬öDec (the wider axis range sets the span for both, centered on the data) and a **square** plot so **1ŌĆ│ horizontally equals 1ŌĆ│ vertically**. Under each chart, the first line is **╬Ż|╬östep|** on each sky axis (**╬öRA ŌĆ”ŌĆ│ ┬Ę ╬öDec ŌĆ”ŌĆ│** vs the targetŌĆÖs first solved frame; optional detector or **Ōēł** pixel equivalents as below). If any **suspect tracking jumps** were identified (same rule as the dither effectiveness score: move Ōēź max(300ŌĆ│, 5├Ś median dither |╬ö|)), an **amber subtext** below the total lists how many intervals were excluded and the discounted ╬Ż|╬öRA| / ╬Ż|╬öDec| (and detector px) so the older "everything summed" number can still be reconstructed. A second **Typical dither** line shows the **median |╬ö|** across assessed dithers (with detector px when the plate scale is known) so a single huge jump cannot hide the typical session behaviour. Each target also includes an **analytics** strip. **Drift / walking-noise risk** is an advisory color box that looks at intervals without logged dither or center commands. From 0.8.24 the advisory is split into two sub-tiers, each shown as its own chip, with the surrounding box tone driven by the worse of the two (Low / Moderate / Caution):

- **Star shape** ŌĆö about per-exposure motion only. Thresholds are sized to the community "< 1ŌĆō2 px per exposure is acceptable" rule of thumb: Moderate at `Ōēź 1.0 px` (or `Ōēź 2.0ŌĆ│`) per exposure, Caution at `Ōēź 2.0 px` (or `Ōēź 4.0ŌĆ│`). Highly directional drift can nudge the level up but only when the per-exposure motion also clears a small floor (`Ōēź 0.3 px` / `0.8ŌĆ│` for LowŌåÆModerate, `Ōēź 0.5 px` / `1.5ŌĆ│` for ModerateŌåÆCaution), so a visually tame trace cannot flag Caution purely on consistency.
- **Walking noise** ŌĆö about correlated FPN across the stacked image. Driven by a new **dither headroom ratio** = `median observed dither |╬ö| (px) / median cumulative drift between consecutive dithers (px)`. The advisory flags Caution when `direction consistency Ōēź 0.75 AND headroom < 2├Ś AND between-dither drift Ōēź 0.5 px`, Moderate when `consistency Ōēź 0.5 AND headroom < 4├Ś AND Ōēź 0.25 px`, Low otherwise. When a target has fewer than two logged dithers, the walking-noise sub-tier falls back to the worst short-window (5/10/15 frame) drift check used in earlier builds. FITS plate scale and FITS exposure duration sharpen both sub-tiers; everything is still advisory and not a hard verdict on the data.

The hint lines under the chips now read:

- **Star-shape hint:** estimated `ŌĆ”ŌĆ│` drift during a typical exposure (`Ōēł ŌĆ” px`), with a reminder that round stars typically tolerate `< 1ŌĆō2 px` of motion per exposure.
- **Walking-noise hint:** dither headroom `ŌĆ”├Ś` (median dither magnitude vs cumulative drift between dithers ŌĆö `ŌĆ”ŌĆ│ ┬Ę Ōēł ŌĆ” px between dithers`). Higher is safer. When no logged dithers were available, the hint falls back to the worst short-window drift line.

The advisory `Detail` line now names which signal drove the headline (e.g. *"Walking-noise signal drives the headline: dither headroom 1.5├Ś with 87% directional drift between dithers."*), so a Caution headline never disagrees with what stars in single subs look like. **Recommendations** are rewritten around the two sub-tiers: star-shape Caution suggests shorter exposures and polar alignment / guiding; walking-noise Caution suggests increasing **Mount Dither Pixels** in NINA, dithering more often, and addressing field drift (polar alignment / differential flexure). When the walking-noise sub-tier is Caution but star-shape is Low, an info line is added so users do not chase a star-shape problem that is not there. If logged dithers repeated the same direction as the natural drift, an extra recommendation calls that out because randomization is not breaking the FPN pattern. A Moderate walking-noise verdict with `headroom < 3├Ś` also emits an info recommendation that a larger dither widens the margin. **Drift rate** shows net arcseconds per minute (and an approximate pixel/min value when FITS plate scale is available). **Dither effectiveness** counts assessed logged dithers, median measured movement, weak dithers, and repeated-direction dithers. Very large outlier dither intervals are treated as **suspect tracking** and excluded from those dither statistics; the strip reports how much absolute RA/Dec was discounted. **Center-after-drift** counts center events as helpful, partial, or ineffective by comparing pre-center drift to residual drift over the next few frames. A compact **Session quality timeline** labels intervals as stable, drifting, dither recovery, or center recovery. **Settings hints from this session** are advisory only; they summarize patterns such as walking-noise drift risk, weak dithers, high drift rate, repeated dither direction, suspect tracking intervals, or center events that did not reduce residual drift.

New reports embed a structured JSON data block in the HTML. This is what **Compare saved reports** reads for before/after comparisons, so old reports without that block need to be regenerated before they can be compared.

## Technical notes

- **LIGHT filter:** After resolving paths from the log, SeeDrift reads each FITS/XISF primary header and skips frames whose **IMAGETYP** / **OBSTYPE** indicate calibration (bias/dark/flat) when those keywords are present ŌĆö same as earlier builds. File extensions **.fits**, **.fit**, **.fts**, and **.xisf** are accepted from the log.
- **Minimum exposures gate:** After LIGHT frames are identified from FITS headers, SeeDrift checks whether **any single OBJECT** has at least **Minimum exposures per target** LIGHT frames in that run. If **none** do, it **does not plate-solve** (nothing useful would chart). During solving, if plate failures make it **impossible** for any OBJECT to reach that many **successful** solves (even if every remaining frame succeeded), SeeDrift **skips the remaining solves** for that run so you are not stuck solving hundreds of frames that cannot change the outcome. If solves finish (or stop early) but **no target** has enough **successful** solves for that threshold, SeeDrift **still skips correlating NINA sequencer logs** into those frames (saves time on large log sets). The night HTML and **Complete ŌĆö** status explain both cases.
- **Stop vs previous session report:** Stop restricts **Saved image to ŌĆ”** lines to **[arm, disarm]** by log timestamp. Previous session report uses **the entire chosen log file**.
- **Logs folder:** Stop works from **`%LocalAppData%\NINA\Logs\*.log`** only ŌĆö fast compared to walking the image tree. It **pre-filters** by the `yyyyMMdd-HHmmss-ŌĆ”` prefix in each log file name (local dates around arm/disarm, plus one day before the earlier date). If **no** files match that filter, SeeDrift reads **all** logs and logs a warning (covers unusual file names). In practice you often get **one log file per imaging session**, but NINA may **start a new file** when it rotates logs, so one session can span **two** files. The night HTML lists **only** logs that had at least one in-window **Saved image to ŌĆ”** line for that run.
- **Solve failures:** Frames that do not solve are skipped for the trace (implementation logs errors).
- **Jump detection:** Runs on the solved-sample sequence where applicable.
